# 🛸 SAM - Sub-Agent-MCP OS

> **하나의 Claude가, 여러 개의 Claude를 부린다.**
> 완전히 독립된 AI 세션을 새 터미널 탭에 띄우고, 파일 기반 인박스로 결과를 모으는 멀티 에이전트 오케스트레이션 MCP 서버.

> ⚠️ 이 저장소는 **학습/연구용 토이 프로젝트**입니다. "Claude를 어떻게 하면 더 잘 쓸 수 있을까"가 궁금해서 직접 부딪혀 본 기록이에요. 
> 프로덕션용이 아닙니다.

> 🧪 **TL;DR** - "여러 개의 Claude를 병렬로 돌리고 결과를 한곳에 모으면 어떨까?"라는 궁금증에서 출발해, 
> **완전 독립 세션 spawn + 중앙 인박스** 구조를 가장 작은 형태로 설계, 구현했다. 만들며 **MCP 서버 작성, 멀티 에이전트 통신(파일 기반 IPC), 
> 서브에이전트 프롬프트 설계, 터미널 자동화의 함정**을 손으로 익혔다.

## 📖 왜 만들었나

Claude로 일하다 보니 한 가지 생각이 생겼다.

> **AI를 잘 쓴다는 건 "얼마나 많이 시키느냐"가 아니라 "어떻게 일을 쪼개고 위임하고 검증하느냐"의 문제다.**
> 그렇다면 개발자의 역할도 *코드 작성자* 에서 *작업을 분해하고 위임하고 검증하는 오케스트레이터* 로 옮겨가야 한다.

한 세션에서 혼자 다 시키는 데는 한계가 보였다. 그래서 머리로만 굴리던 생각을 손으로 확인하고 싶어졌다 -
**"완전히 독립된 AI 세션을 여러 개 병렬로 띄우고, 결과를 중앙 인박스로 모은다"** 는 구조를 내가 만들 수 있는 가장 작은 형태로 직접 구현해 봤다. 
이 저장소가 그 결과물이다.

### 내가 풀고 싶었던 문제

처음엔 의문이 있었다. Claude Code엔 이미 서브에이전트(`Agent` 도구)가 있다. 한 세션 안에서 병렬로 일도 시킬 수 있는데, 
**굳이 새 터미널 탭을 띄울 이유가 있나?**

직접 만들어 굴려보면서 답을 찾았다. 내장 서브에이전트는 두 가지가 아쉬웠다.

- 부모와 **같은 세션 생명주기**에 묶인다. 부모가 끝나면 자식도 같이 사라진다.
- 작업 과정이 **블랙박스**다. 최종 요약만 돌아오고, 중간에 내가 끼어들기 어렵다.

그래서 다른 길을 택했다.

> **각 자식을 "완전히 독립된 `claude` 세션"으로, 새 iTerm 탭에 띄운다.**
> 부모와 컨텍스트를 공유하지 않으니 부모의 컨텍스트가 오염되지 않고,
> 무엇보다 **내가 각 탭에서 일하는 걸 눈으로 보고 직접 끼어들 수 있다.**

결국 이 한 가지 차이 - **"내가 과정을 보고 끼어들 수 있느냐"** - 가 전체 설계를 끌고 갔다.

### "한 세션 내 서브에이전트"와 뭐가 다른가

| | **SAM (이 프로젝트)** | 내장 `Agent` 도구 |
|---|---|---|
| 실행 형태 | 새 iTerm 탭의 **독립 `claude` 세션** | 같은 프로세스 내 서브에이전트 |
| 실시간 관찰/개입 | ✅ 탭마다 눈으로 보고 끼어듦 | ❌ 최종 요약만 (블랙박스) |
| 부모와 독립된 생명주기 | ✅ 부모가 죽어도 자식은 계속 | ❌ 부모 끝나면 같이 종료 |
| 대화형 다회차 릴레이 | ✅ `send_message`로 계속 지시 | △ 제한적 |
| 신뢰성 | △ 터미널 자동화라 타이밍에 민감 | ✅ API 호출이라 결정적 |

**솔직히 말하면**, 그냥 "작업 쪼개서 결과만 받기"가 목적이라면 내장 `Agent`/`Workflow`가 더 우월하다.
SAM이 의미 있는 건 **"사람이 지켜보며 대화로 조종하는, 오래 사는 Claude 팀룸"** 이 필요할 때다.
그리고 무엇보다 - 이 구조를 **직접 만들어보며 이해하는 것** 자체가 내 목적이었다.

> 🧭 **언제 무엇을 쓰나 (내 판단):**
> **결과만 필요하면 내장 `Agent`**, **과정을 지켜보며 조종하거나 오래 살려둘 거면 SAM.**
> 도구의 화려함이 아니라 *상황에 맞는 선택*이 중요하다고 생각해서, 내가 만든 것조차 "이게 항상 정답은 아니다"라고 적어둔다.

## 🏗️ 아키텍처

```
                         ┌────────────────────────────┐
                         │   부모 Claude (오케스트레이터)  │
                         │   delegate / check / send  │
                         └────────────┬───────────────┘
                                      │ MCP (stdio)
                         ┌────────────▼───────────────┐
                         │   sam_server.py (MCP 서버)  │
                         └────────────┬───────────────┘
            osascript(AppleScript)    │     파일 read/write
        ┌─────────────┬───────────────┼───────────────┐
        ▼             ▼               ▼               │
   ┌─────────┐  ┌─────────┐     ┌─────────┐           │
   │ iTerm 탭 │ │ iTerm 탭 │     │ iTerm 탭 │   ...    │
   │ 자식 #1  │  │ 자식 #2  │     │ 자식 #3  │           │
   │ (claude)│  │ (claude)│     │ (claude)│           │
   └────┬────┘  └────┬────┘     └────┬────┘           │
        │ report     │ report        │ report         │
        ▼            ▼               ▼                ▼
   ┌────────────────────────────────────────────────────┐
   │            .sam_inbox/  (중앙 메시지 인박스)            │
   │   {id}.json     {id}_prompt.md     {id}_context.md │
   └────────────────────────────────────────────────────┘
```

### 프로젝트 구조

```text
sub-agent-mcp/
├── sam_server.py          # MCP 서버 엔트리포인트 - 도구 5개 등록 + stdio 구동
├── sam_config.py          # 경로/타이밍 설정 (BASE_DIR, INBOX_DIR, BOOT_DELAY …)
├── setup_sam.sh           # venv 생성 + MCP 전역(user scope) 등록 스크립트
├── tools/                 # 도구별 모듈
│   ├── delegate.py        #   delegate_to_sam - 새 iTerm 탭에 독립 자식 세션 생성
│   ├── report.py          #   report_result   - 자식이 결과를 인박스에 기록
│   ├── inbox.py           #   check_inbox     - 자식들 완료 여부 폴링/수집
│   ├── message.py         #   send_message    - 대기 중 자식에게 추가 지시(릴레이)
│   ├── close.py           #   close_session   - 탭 종료 + 인박스 파일 정리
│   └── store.py           #   (공용) 세션 JSON 원자적 읽기/쓰기 헬퍼
├── tests/
│   └── test_smoke.py      # iTerm/AppleScript 없이 핵심 로직(원자적 저장, 인젝션 차단) 검증
└── .sam_inbox/            # 런타임에 자동 생성되는 중앙 인박스
    ├── {id}.json          #   세션 상태/결과
    ├── {id}_prompt.md     #   자식에게 건넨 지시문
    └── {id}_context.md    #   (선택) rag_context
```

> `venv/`와 `.sam_inbox/`는 실행 중 생성되며 `.gitignore`로 제외된다. 인박스 위치는 클론한 폴더 기준이고, `SAM_HOME` 환경변수로 옮길 수 있다.

### 동작 흐름

1. **위임** - 부모가 `delegate_to_sam(task)` 호출 -> 서버가 ① 인박스에 세션 JSON 생성, ② 지시문을 `{id}_prompt.md`로 저장, ③ AppleScript로 새 iTerm 탭을 열어 `claude` 실행, ④ 자식에게 *"그 파일을 읽고 따르라"* 한 줄만 전송.
2. **작업** - 자식 Claude가 독립적으로 임무 수행.
3. **보고** - 자식이 `report_result(session_id, result)` 호출 -> 세션 JSON의 `status`가 `SUCCESS`로, `result`가 채워짐.
4. **수집** - 부모가 `check_inbox([ids])`로 폴링 -> 완료된 것부터 결과 회수.
5. **(선택) 릴레이** - 부모가 `send_message(id, "추가 지시")`로 대기 중인 자식에게 후속 작업 전달.
6. **정리** - `close_session(id)`로 탭 종료 + 인박스 파일 삭제.

> 💡 **왜 지시문을 "파일로 저장하고 한 줄만 보내나?"**
> 처음엔 프롬프트 전체를 탭에 그대로 타이핑하게 만들었는데, 자꾸 깨졌다. 알고 보니 iTerm의 `write text`는 **개행을 Enter로 전송**한다. 여러 줄 프롬프트가 줄마다 따로 제출되고 있었던 것. 그래서 전체 지시문은 `{id}_prompt.md`에 쓰고, 자식에겐 개행 없는 한 줄(`그 파일 읽어라`)만 보내도록 바꿨다. 덤으로 따옴표, 백틱 같은 문자가 AppleScript로 새는 인젝션 문제도 같이 사라졌다.

## 🤔 설계하며 내린 선택들

토이지만, 매 결정마다 "왜 이게 더 나은가, 무엇을 포기하는가"를 따져봤다. 대안을 버린 이유가 곧 설계다.

| 결정 | 택한 것 | 버린 대안 | 이유                                                                 |
|------|---------|-----------|--------------------------------------------------------------------|
| **자식 실행 방식** | 독립 `claude` 세션 (새 탭) | 내장 서브에이전트 | 부모 컨텍스트 오염 방지 + **사람이 보고 끼어들 수 있음** + 부모와 독립된 생명주기                 |
| **부모↔자식 통신** | 파일 기반 인박스(`.sam_inbox/*.json`) | 소켓 / 메시지큐 | 의존성 0, **사람이 직접 열어볼 수 있는 투명함**, 프로세스가 죽어도 상태가 디스크에 남음. (대가: 폴링 필요) |
| **완료 감지** | 부모가 `check_inbox`로 **폴링** | 자식이 부모에게 푸시 | 부모가 수집 주도권을 가짐 + 구현 단순. 토이 규모에선 폴링 비용이 무시할 만함                      |
| **지시문 전달** | 파일에 쓰고 "읽어라" 한 줄 | 프롬프트 전체를 타이핑 | `write text`의 개행=Enter 문제 + 인젝션을 **한 번에** 차단                       |
| **결과 보고** | 자식이 `report_result` 도구 호출 | 터미널 화면 스크래핑 | 구조적, 결정적. 화면 파싱은 깨지기 쉽고 자식의 사고 과정과 결과가 뒤섞임                         |
| **세션 저장 위치** | 클론 위치 자동 인식(+`SAM_HOME`) | 고정 경로 하드코딩 | 누가 받아써도 동작하게. 하드코딩은 "내 컴에서만 됨"의 지름길                                |

## 🧠 서브에이전트를 '말 잘 듣게' 만들기 (프롬프트 설계)

이 프로젝트에서 가장 오래 고민한 건 코드가 아니라 **자식 Claude에게 보내는 프롬프트**였다. 독립 세션은 부모의 맥락을 1도 모르는 채 깨어나기 때문에, 첫 지시문이 곧 그 세션의 운영체제다. 내가 박아 넣은 장치들:

- **자기 ID를 알려준다.** 자식은 자기 `session_id`를 모르면 `report_result(session_id, ...)`로 보고할 수가 없다. 그래서 지시문에 ID를 명시적으로 박는다. *"자식이 자기 자신을 식별할 수단을 부모가 쥐여줘야 한다"* 는 걸 보고가 안 돌아오는 걸 보고서야 깨달았다.
- **"끝나도 사라지지 마라"를 명시한다.** Claude는 작업을 마치면 자연스럽게 대화를 닫으려 한다. 그러면 릴레이(`send_message`)가 불가능하다. 그래서 *"보고 후에도 종료하지 말고 추가 지시를 대기하라"* 를 규칙으로 넣었다.
- **우선순위 충돌을 미리 정리한다.** 자식이 작업 디렉토리의 `CLAUDE.md`와 위임 지시 사이에서 헷갈리지 않도록, *"CLAUDE.md가 있어도 본 지시문이 최우선"* 임을 첫 줄에서 못 박는다.
- **컨텍스트는 격리하되, 필요한 지식은 주입한다.** 부모 맥락을 공유하지 않는 게 핵심이지만, 팀 규칙처럼 꼭 필요한 건 `rag_context`로 별도 파일을 만들어 "작업 전에 이것부터 읽어라"로 건넨다. **컨텍스트는 양이 아니라 질이 결과를 결정한다**고 보고, 꼭 필요한 것만 골라 넣는 데 신경 썼다.

> 한 줄 요약: **부모의 컨텍스트를 지키는 것 = 부모의 장기적 판단력을 지키는 것.** 자식에게 일을 떼어내는 진짜 이유는 "병렬"이 아니라 "오케스트레이터의 머리를 깨끗하게 유지"하기 위함이라는 걸, 만들면서 이해했다.

## 🔧 만들며 부딪힌 것들 (그리고 어떻게 고쳤나)

튜토리얼엔 안 나오는, 직접 굴려보며 터진 것들:

- **멀티라인 프롬프트가 조각나 제출됨** - `write text`가 개행을 Enter로 보내서, 한 프롬프트가 여러 메시지로 쪼개졌다. -> 지시문을 파일로 저장하고 한 줄만 전송.
- **타이밍 레이스** - `claude`가 부팅되기 전에 프롬프트가 빈 셸에 입력됐다. -> 부팅 대기(`BOOT_DELAY`)를 설정값으로 분리(완벽한 해법은 아니라 한계 섹션에 솔직히 적어둠).
- **동시 파일 읽기/쓰기 깨짐** - 자식이 JSON을 *쓰는 중* 부모가 읽으면 `JSONDecodeError`로 폴링 전체가 죽었다. -> **원자적 쓰기**(임시파일 -> `os.replace`) + 재시도 읽기.
- **AppleScript 인젝션** - task에 따옴표, 백틱이 들어가면 스크립트가 깨지거나 실행이 샜다. -> 파일 경유 전달로 task가 애초에 스크립트에 닿지 않게 함.
- **"다 끝났다"는 거짓 신호** - 세션 파일이 없는 걸 완료로 착각하던 폴링 로직을 success/pending을 분리 집계하도록 고침.

## 🧰 제공 도구 (MCP Tools)

| 도구 | 설명 |
|------|------|
| `delegate_to_sam(task, rag_context="", working_dir="")` | 독립 자식 세션을 새 iTerm 탭에 생성하고 작업 위임. `rag_context`를 주면 팀 규칙/지식을 별도 파일로 자식에게 먼저 읽힘. `working_dir`로 자식이 일할 디렉토리 지정. -> 세션 ID 반환 |
| `report_result(session_id, result, summary="")` | (자식이 호출) 결과를 부모에게 구조화해 보고 |
| `check_inbox(session_ids)` | 자식들의 완료 여부를 폴링해 결과 수집. 응답 없는 세션은 경과 시간으로 stale 경고 |
| `send_message(session_id, message)` | 대기 중인 자식에게 추가 지시(릴레이) 전달 |
| `close_session(session_id)` | 탭 종료 + 인박스 파일 정리 |

**구현 디테일 몇 가지:**

- **탭을 ID로 정확히 지목한다.** `delegate_to_sam`은 새 탭을 열 때 iTerm이 부여한 고유 `iterm_id`를 받아 세션 JSON에 함께 저장한다. 이 ID가 있어야 나중에 `send_message`, `close_session`이 **수많은 탭 중 그 자식의 탭만** 골라 지목할 수 있다.
- **추가 지시는 화면을 안 뺏고 전달된다.** `send_message`는 AppleScript에서 `activate`(앱 전면 전환)나 포커스 이동 없이 대상 세션에 바로 텍스트를 쓴다. 내가 다른 창에서 작업 중이어도 자식 탭이 백그라운드에서 조용히 지시를 받는다.
- **결과 보고는 화면이 아니라 파일로 간다.** 자식은 사람에게 메시지를 띄우는 게 아니라 인박스 JSON의 `status`를 `SUCCESS`로 바꾸고 `result`를 기록한다. 그래서 부모가 자기 페이스로 `check_inbox` 폴링만 하면 된다(비동기 수집).

### 세션 JSON 스키마 (`.sam_inbox/{id}.json`)

```json
{
  "session_id": "9dd939b5",
  "status": "RUNNING | SUCCESS | SPAWN_FAILED",
  "task": "위임된 작업 설명",
  "result": "자식이 보고한 결과",
  "iterm_id": "iTerm2 세션 ID",
  "created_at": "2026-06-22T10:00:00+09:00",
  "closed_at": "",
  "cwd": "/Users/me/work/some-repo",
  "git_branch": "feature/ABC-123"
}
```

## 🚀 시작하기 (설치 가이드)

### 요구사항

- **macOS** + **[iTerm2](https://iterm2.com/)** (AppleScript 자동화를 쓰므로 필수)
- **[Claude Code CLI](https://docs.claude.com/claude-code)** (`claude` 명령이 PATH에 있어야 함)
- **Python 3.10+**
- (선택) `git` - 자식 작업 디렉토리의 브랜치를 메타데이터로 기록

### 1. 원하는 위치에 클론

이 저장소는 **어디에 두든 동작한다.** 경로가 고정돼 있지 않고, 클론한 위치를 자동으로 루트로 잡는다.
원하는 경로를 골라 클론하자. (아래 `<클론할 경로>` 자리에 본인이 원하는 위치를 넣으면 된다.)

```bash
git clone https://github.com/minBuild/sub-agent-mcp.git <클론할 경로>
# 예) git clone https://github.com/minBuild/sub-agent-mcp.git ~/dev/sub-agent-mcp
cd <클론할 경로>
```

### 2. 셋업 스크립트 실행

```bash
chmod +x setup_sam.sh
./setup_sam.sh
```

`setup_sam.sh`가 하는 일:
1. **스크립트가 있는 폴더(= 방금 클론한 위치)** 를 SAM 루트로 인식
2. Python 가상환경(`venv`) 생성 + `mcp` 패키지 설치
3. MCP 서버를 **전역(user scope)** 으로 등록 (경로는 클론 위치 기준 자동 설정):
   ```bash
   claude mcp add --scope user sub-agents-mcp -- \
     <클론한 경로>/venv/bin/python <클론한 경로>/sam_server.py
   ```

> 🛠️ **인박스 위치를 따로 두고 싶다면?** `SAM_HOME` 환경변수로 덮어쓸 수 있다.
> 지정하지 않으면 클론한 폴더를 그대로 쓴다.
> ```bash
> export SAM_HOME="/원하는/세션/저장/경로"
> ```

### 3. 권한 허용

처음 `claude`를 실행하면 SAM 도구 사용 권한 팝업이 뜬다. 검토 후 허용하면 된다.
미리 허용해두려면 프로젝트 `.claude/settings.json`에 추가:

```json
{
  "permissions": {
    "allow": [
      "mcp__sub-agents-mcp__delegate_to_sam",
      "mcp__sub-agents-mcp__check_inbox",
      "mcp__sub-agents-mcp__send_message",
      "mcp__sub-agents-mcp__close_session",
      "mcp__sub-agents-mcp__report_result"
    ]
  }
}
```

### 4. 써보기

부모가 될 `claude` 세션에서 자연어로 지시하면 된다:

```
> 이 3개 모듈을 병렬로 리뷰해줘. 각각 독립 세션으로 띄워서.
```

그러면 부모 Claude가 `delegate_to_sam`을 3번 호출 -> iTerm 탭 3개가 열리고, 각 탭의 Claude가 독립적으로 작업한다. 부모는 `check_inbox`로 결과를 모아 정리해 준다.

> ⏱️ **타이밍 주의:** 자식 탭에서 `claude`가 부팅되는 시간을 `sam_config.py`의 `BOOT_DELAY`(기본 6초)로 기다린다. 머신이 느려 프롬프트가 빈 셸에 입력되면 이 값을 늘리자.

## 🚨 문제 해결 (Troubleshooting)

**`./setup_sam.sh: permission denied`**
실행 권한이 없는 경우다. `chmod +x setup_sam.sh` 후 다시 실행.

**다른 디렉토리로 가면 SAM 도구가 안 보인다**
`setup_sam.sh`는 `--scope user`로 전역 등록하므로 어디서나 보여야 정상이다. 안 보이면 등록 상태부터 확인:
```bash
claude mcp list   # sub-agents-mcp 가 있는지
```
구버전을 로컬 스코프로 등록했던 잔재가 있으면, 그걸 제거하고 `setup_sam.sh`로 다시 등록한다.

**자식 탭은 열렸는데 프롬프트가 빈 셸에 입력된다 / `claude`가 안 떠 있다**
`claude` 부팅이 `BOOT_DELAY`(기본 6초)보다 오래 걸린 경우다. `sam_config.py`의 `BOOT_DELAY` 값을 늘려라.

**`osascript` / iTerm 자동화가 막힌다 (탭이 안 열림)**
macOS의 자동화 권한 문제일 수 있다. **시스템 설정 -> 개인정보 보호 및 보안 -> 자동화**에서 터미널/iTerm이 iTerm을 제어하도록 허용했는지 확인.

## ⚠️ 알려진 한계 (솔직하게)

- **macOS + iTerm2 전용.** AppleScript에 의존하므로 Linux/Windows/다른 터미널에선 안 된다.
- **타이밍에 민감.** "claude가 준비됐는지"를 감지하는 게 아니라 고정 시간을 기다린다(`BOOT_DELAY`). 완전한 견고성을 원하면 탭 화면을 폴링해 준비 신호를 감지해야 하는데, 토이 범위를 넘어 생략했다.
- **결과 회수는 자식의 협조에 의존.** 자식이 `report_result`를 안 부르면 부모는 계속 폴링한다. (`check_inbox`가 stale 세션을 경고는 해준다.)
- **단일 사용자/로컬 전제.**

## 🙇 만들며 배운 것

작게 만들었지만 얻은 건 작지 않았다.

- **MCP 서버를 처음부터 작성**하며 도구 등록, stdio 통신 구조를 이해했다.
- **에이전트 간 통신을 직접 설계**하며, 화려한 메시지큐 대신 "파일 + 폴링"이 토이 규모엔 왜 충분한지 체감했다.
- **서브에이전트 프롬프트 설계**가 코드만큼, 때론 코드보다 중요하다는 걸 배웠다.
- 무엇보다 - **내장 기능을 그대로 쓰는 것과, 직접 만들어 그 한계까지 아는 것은 다르다**는 걸 손으로 확인했다.
